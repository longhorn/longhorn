# SPDK v26.05 Bump for Longhorn V2

## Summary

This proposal covers bumping Longhorn's SPDK dependency from the current Longhorn-maintained pin (based on SPDK v25.09 / v26.01-era forks) to **upstream SPDK v26.05**, re-porting Longhorn's existing SPDK delta (raid1, lvol/blob features used by the V2 engine), and carrying a small set of additive fixes and adaptations required for V2 (including RDMA and iobuf-backed NVMf transports).

Production experience on large V2 fleets showed markedly better stability on v26.05 versus earlier SPDK lines (fewer reactor wedges, connect/disconnect storms, and cascading instance-manager failures). This LEP proposes adopting that base deliberately, with a clear inventory of Longhorn-carried vs additive patches.

## Motivation

### Goals

- Pin Longhorn V2's `spdk_tgt` build to **SPDK v26.05.x**.
- Re-port the Longhorn-owned SPDK patches currently carried on older bases (raid1 single-base/grow, lvol/blob xattrs, snapshot checksums, range/deep/shallow copy, fragmap, interrupt-mode TCP support chain, and related Longhorn fixes) onto v26.05.
- Adopt v26.05 behavioral changes that Longhorn must handle in Go:
  - NVMf buffers drawn from the shared **iobuf** pool (transport create options / cache caps).
  - `memory_domains.dma_device_type` as a **string** in JSON-RPC responses.
  - genrpc / schema expectations for Longhorn-added RPCs.
  - Support for `--wait-for-rpc` bring-up (`framework_start_init` / wait).
- Document a minimal additive patch set needed for RDMA/RoCE and link correctness on v26.05.
- Update `dep-versions`, instance-manager image build, and go-spdk-helper wrappers accordingly.

### Non-goals

- Re-submitting the large historical crash-consistency / UAF experiment series from older forks unless individually reproduced on v26.05.
- Reworking Longhorn control-plane rebuild into SPDK (out of scope).
- Changing default cluster size or other production constants as part of the bump (orthogonal settings discussions).

## Proposal

### User Stories

#### Story 1

As a Longhorn maintainer, I want a single clear SPDK base (v26.05) plus an explicit re-port of Longhorn's required patches so V2 builds are reproducible and reviewable.

#### Story 2

As a V2 operator, I want the instance manager to size iobuf pools and transport caches correctly after the bump so NVMf/RDMA bring-up does not OOM or stall under reactor-scaled defaults.

### User Experience In Detail

- Operators upgrading Longhorn to a release that includes this bump get a new instance-manager image built against SPDK v26.05.
- No new user-facing setting is *required* for the bump itself; existing v2 settings continue to apply.
- Clusters using RDMA (separate LEP) benefit from RDMA-related additive SPDK fixes included in the pin.
- Optional: shallow-copy pipeline depth (separate LEP) may land in the same SPDK pin or immediately after.

### API changes

- No Longhorn CRD API required solely for the version bump.
- go-spdk-helper JSON types/RPCs updated for v26.05 response shapes and transport/iobuf options.
- Engine may set iobuf options and framework init around `spdk_tgt` start when using `--wait-for-rpc`.

## Design

### Implementation Overview

#### Layer 1 — Longhorn SPDK delta on v26.05

Re-port the Longhorn team's existing `longhorn-v25.09` (and related) content onto `v26.05.0`, resolving conflicts with upstream blob/lvol changes. This is Longhorn-owned code, not a new feature proposal in this LEP.

#### Layer 2 — Additive patches (small)

Representative additive fixes validated with the bump (exact commit list lives on the SPDK branch):

- NVMe/RDMA: NULL-guard `rdma_qp` in `ctrlr_get_memory_domains` (crash under RDMA).
- mlx5: force `crc32c_supported=false` when firmware advertises CRC32C but sig-mkey pool allocation fails at scale.
- NVMe: async-connect poller owned by the qpair + re-entrancy safety (UAF / re-entry).
- Build: declare `lib/nvme` dependency on `thread` for the async-connect poller after v26.05 sock_posix changes.
- Port glue: align Longhorn RPC / range-shallow-copy decoders with v26.05 genrpc autogen/lint.
- Optional follow-on: shallow-copy pipelining (see shallow-copy pipeline LEP).

#### Go / IM adaptations

- **go-spdk-helper**: `dma_device_type` string decode; iobuf / nvmf transport option wrappers; `FrameworkStartInit` / `FrameworkWaitInit`; attach timeout classifiers as needed.
- **longhorn-spdk-engine**: budget-aware iobuf pool sizing for v26.05's shared iobuf + per-poll-group caches (defaults conservative; overridable via env); call framework init when the tgt starts with `--wait-for-rpc`.
- **dep-versions / instance-manager**: pin SPDK SHA/tag; SLES/python venv build fixes for v26.05; RDMA link asserts when RDMA is enabled.

### Test plan

- Build IM image against the new pin; verify `spdk_tgt` starts with TCP (and RDMA when enabled).
- V2 smoke: create/attach/detach, rebuild, snapshot, backup expose.
- Stress: IM restart under load; multi-volume attach; RDMA path if hardware available.
- Regression: confirm Go client parses `bdev_get_bdevs` memory domains after the `dma_device_type` change.
- Memory: confirm iobuf sizing floors prevent unbounded per-pg cache growth on high reactor counts.

### Upgrade strategy

- Standard IM image rollout (V2 requires detach/reattach / IM recycle; no live migration).
- Roll TCP-only clusters first if desired; enable RDMA labels afterward.
- Keep the previous SPDK pin recoverable via image rollback.

## Note

Stability gains observed when moving fleets to v26.05 appear dominated by the SPDK version itself. Remaining bugs on latest Longhorn + v26.05 should be filed as individual GitHub tickets with reproducers rather than bundled into a catch-all crash-consistency LEP.
